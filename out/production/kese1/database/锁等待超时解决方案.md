# 锁等待超时问题解决方案

## 问题描述
新增水质检测记录时出现错误：
```
Lock wait timeout exceeded; try restarting transaction
```

## 根本原因

### 1. 触发器性能问题
原触发器 `trg_auto_warning` 在批量插入时，对每一行检测详情都会执行：
- 查询 `water_quality_records` 表获取 `plant_id`
- 查询 `water_quality_standards` 表获取 `item_name`  
- 如果不合格，插入 `warnings` 表

**问题点：**
- 批量插入 8 个检测项 = 触发 8 次 = 16 次查询 + 可能的 8 次插入
- 所有操作在同一事务中，持有锁的时间过长
- 原触发器还有冗余的 `SELECT record_id INTO v_record_id` 查询（完全没必要）

### 2. 可能的外部因素
- 其他未提交的事务占用了资源
- 数据库锁等待时间设置过短（默认 50 秒）
- 并发访问同一张表

## 解决方案

### 方案一：优化触发器（推荐优先尝试）⭐

**步骤：**

1. **打开 MySQL 命令行或图形化工具**

2. **执行修复 SQL：**
```sql
-- 查看当前锁等待情况
SHOW PROCESSLIST;

-- 删除旧触发器
DROP TRIGGER IF EXISTS trg_auto_warning;

DELIMITER $$

-- 创建优化后的触发器
CREATE TRIGGER trg_auto_warning
AFTER INSERT ON water_quality_details
FOR EACH ROW
BEGIN
    IF NEW.is_qualified = 0 THEN
        DECLARE v_plant_id INT;
        DECLARE v_item_name VARCHAR(100);
        
        -- 只查询必要字段（移除了冗余的 v_record_id）
        SELECT wr.plant_id INTO v_plant_id 
        FROM water_quality_records wr 
        WHERE wr.record_id = NEW.record_id;
        
        SELECT ws.item_name INTO v_item_name 
        FROM water_quality_standards ws 
        WHERE ws.standard_id = NEW.standard_id;
        
        -- 确保数据有效才插入预警
        IF v_plant_id IS NOT NULL AND v_item_name IS NOT NULL THEN
            INSERT INTO warnings (record_id, plant_id, warning_type, warning_level, title, content)
            VALUES (NEW.record_id, v_plant_id, 'quality', 'high', 
                    CONCAT('水质检测异常 - ', v_item_name),
                    CONCAT('检测项 "', v_item_name, '" 不合格，测量值：', NEW.measured_value));
        END IF;
    END IF;
END$$

DELIMITER ;

-- 验证触发器
SHOW TRIGGERS LIKE 'water_quality_details';
```

3. **重启应用并测试**

### 方案二：调整数据库锁等待时间（临时方案）

**查看当前锁等待时间：**
```sql
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';
```

**临时增加锁等待时间（秒）：**
```sql
SET GLOBAL innodb_lock_wait_timeout = 120;  -- 增加到 120 秒
```

**永久修改：** 在 `my.ini` 或 `my.cnf` 中添加：
```ini
[mysqld]
innodb_lock_wait_timeout=120
```

---

### 方案三：清理未提交的事务（紧急处理）

**查看正在运行的事务：**
```sql
SHOW ENGINE INNODB STATUS;
```

**查找并终止长时间运行的事务：**
```sql
-- 找到 problematic 的线程 ID
SHOW PROCESSLIST;

-- 终止事务（替换 YOUR_THREAD_ID）
KILL YOUR_THREAD_ID;
```

### 方案四：修改 Java 代码逻辑（彻底解决）

如果上述方案都不能解决问题，考虑修改代码策略：

**思路：** 将预警生成从触发器改为应用层处理

**修改文件：** `WaterQualityService.java`

```java
public void addRecord(WaterQualityRecord record, List<WaterQualityDetail> details) throws Exception {
    // ... 原有代码保持不变 ...
    
    // 提交事务后，异步处理预警
    com.ruralwater.util.DBUtil.commit(conn);
    
    // 在事务外处理预警，避免长时间持有锁
    if (hasUnqualifiedItem(details)) {
        createWarningsAsync(recordId, details);
    }
}

private boolean hasUnqualifiedItem(List<WaterQualityDetail> details) {
    for (WaterQualityDetail detail : details) {
        if (detail.getIsQualified() != null && detail.getIsQualified() == 0) {
            return true;
        }
    }
    return false;
}

private void createWarningsAsync(Integer recordId, List<WaterQualityDetail> details) {
    // 在新的事务中处理预警，不阻塞主流程
    new Thread(() -> {
        try {
            // 延迟一点执行，让主事务完全提交
            Thread.sleep(100);
            
            Connection conn = DBUtil.getConnection();
            // 获取 plant_id
            PreparedStatement pstmt1 = conn.prepareStatement(
                "SELECT plant_id FROM water_quality_records WHERE record_id = ?");
            pstmt1.setInt(1, recordId);
            ResultSet rs = pstmt1.executeQuery();
            int plantId = 0;
            if (rs.next()) {
                plantId = rs.getInt("plant_id");
            }
            rs.close();
            pstmt1.close();
            
            // 为每个不合格项创建预警
            for (WaterQualityDetail detail : details) {
                if (detail.getIsQualified() == 0) {
                    PreparedStatement pstmt2 = conn.prepareStatement(
                        "SELECT item_name FROM water_quality_standards WHERE standard_id = ?");
                    pstmt2.setInt(1, detail.getStandardId());
                    ResultSet rs2 = pstmt2.executeQuery();
                    String itemName = "";
                    if (rs2.next()) {
                        itemName = rs2.getString("item_name");
                    }
                    rs2.close();
                    pstmt2.close();
                    
                    // 插入预警
                    PreparedStatement pstmt3 = conn.prepareStatement(
                        "INSERT INTO warnings (record_id, plant_id, warning_type, warning_level, title, content) " +
                        "VALUES (?, ?, 'quality', 'high', ?, ?)");
                    pstmt3.setInt(1, recordId);
                    pstmt3.setInt(2, plantId);
                    pstmt3.setString(3, "水质检测异常 - " + itemName);
                    pstmt3.setString(4, "检测项 \"" + itemName + "\" 不合格，测量值：" + detail.getMeasuredValue());
                    pstmt3.executeUpdate();
                    pstmt3.close();
                }
            }
            
            conn.close();
        } catch (Exception e) {
            e.printStackTrace();
        }
    }).start();
}
```

## 排查步骤总结

### 1. 立即检查
```sql
-- 查看是否有未提交的事务
SHOW PROCESSLIST;

-- 查看锁等待状态
SHOW ENGINE INNODB STATUS;
```

### 2. 执行修复
- 运行 `database/fix_lock_timeout.sql`
- 重启应用

### 3. 测试验证
- 添加新的水质检测记录
- 观察是否还出现超时

### 4. 如果仍然超时
- 增加 `innodb_lock_wait_timeout`
- 或者采用方案四，移除触发器逻辑

---

## 预防措施

1. **定期清理长事务**
   ```sql
   -- 查看超过 60 秒的事务
   SELECT * FROM information_schema.INNODB_TRX 
   WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 60;
   ```

2. **优化批量插入**
   - 分批提交（每批 5-10 条）
   - 避免在一个事务中做太多操作

3. **监控数据库性能**
   - 使用慢查询日志
   - 定期检查锁等待情况
